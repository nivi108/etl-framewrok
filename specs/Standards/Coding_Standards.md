# Coding Standards

**Version:** 1.0
**Last Modified:** 2026-07-13
**Depends On:** Project_Architecture.md (v1.0), Repository_Structure.md (v1.0)
**Category:** Standards

## Purpose
Defines the coding conventions all generated PySpark notebooks, SQL scripts, and utility functions must follow. These standards exist so that code generated by different agents (raw_agent, silver_agent, gold_agent, etc.) looks and behaves consistently — as if one developer wrote the entire project, not six independent agents.

## Scope
Covers PySpark style, SQL style, variable naming, function structure, comments, and reusable component usage. Does NOT cover file naming (that's `Naming_Standards.md`).

## PySpark Style

| Rule | Convention |
|---|---|
| Import ordering | Standard library → PySpark → Framework components → Third-party, separated by blank lines |
| DataFrame variables | Descriptive, suffixed with `_df` (e.g., `orders_raw_df`, `customers_cleaned_df`) |
| Column references | Always use `F.col("column_name")`, never bare string column references in transformations — makes column lineage explicit and grep-able |
| Chaining | Chain transformations with one operation per line, aligned by `.` — never write multi-step transformations as a single unreadable line |
| String quoting | Double quotes for strings, single quotes only inside SQL expressions passed as strings |
| Magic numbers | Never — use named constants or config values for thresholds, defaults, retry counts, etc. |
| Type hints | Required on all function signatures — return types and parameter types |

**Example (illustrative):**
```python
from pyspark.sql import functions as F
from framework.components import read_config, logging_component

def load_orders(table_name: str, pipeline_id: str) -> bool:
    config = read_config.read_config(table_name)
    orders_raw_df = (
        spark.read
        .format("jdbc")
        .option("url", connection_url)
        .load()
    )
    return True
```

## SQL Style

| Rule | Convention |
|---|---|
| Keywords | UPPERCASE (`SELECT`, `FROM`, `WHERE`, `MERGE INTO`) |
| Table/column names | lowercase_snake_case |
| Indentation | 4 spaces, consistent across all SQL |
| One clause per line | Each `SELECT`, `FROM`, `WHERE`, `AND`, `JOIN`, `ON` starts a new line |
| Aliases | Always use explicit `AS` keyword, never implicit aliasing |
| Star selects | Never use `SELECT *` in Silver/Gold — always explicit column lists for traceability; `SELECT *` is permitted only in Raw ingestion where the intent is to land all source columns |

**Example (illustrative):**
```sql
MERGE INTO silver_orders AS target
USING staged_orders AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN
    UPDATE SET
        target.amount = source.amount,
        target._processed_at = current_timestamp()
WHEN NOT MATCHED THEN
    INSERT (order_id, amount, _processed_at)
    VALUES (source.order_id, source.amount, current_timestamp())
```

## Variable Naming Conventions

| Context | Convention | Example |
|---|---|---|
| DataFrames | `{entity}_{layer}_df` | `orders_raw_df`, `customers_silver_df` |
| Config objects | `{entity}_config` | `orders_config` |
| Counts | `{entity}_{metric}_count` | `orders_source_count`, `orders_rejected_count` |
| Booleans | `is_` or `has_` prefix | `is_active`, `has_duplicates` |
| Constants | UPPER_SNAKE_CASE | `LARGE_TABLE_THRESHOLD`, `NULL_SENTINEL` |
| Temporary/intermediate | `_tmp_` prefix | `_tmp_deduped_df` |

## Function Standards

| Rule | Convention |
|---|---|
| Single responsibility | One function does one thing — never a function that reads, transforms, and writes |
| Naming | `verb_noun` pattern: `read_config`, `validate_schema`, `compute_hash` |
| Return values | Always explicit return, never rely on implicit None — return a result object or raise |
| Docstrings | Required on every function: one-line summary, parameters, returns, raises |
| Max length | Functions should not exceed ~50 lines; if longer, decompose into sub-functions |

## Reusable Component Usage

| Rule | Rationale |
|---|---|
| Always call shared Components for config reading, logging, auditing, merging, hashing, notification | Never re-implement logic already covered by a Component spec |
| Never import internal implementation details of a Component — only its public function(s) | Prevents coupling to a Component's internal structure, which may change |
| If a notebook needs behavior not covered by any existing Component, flag it as a gap rather than writing one-off logic inline | Maintains the "generate from specs" discipline; one-off logic is a sign the specs need extending |

## Comments

| Rule | Convention |
|---|---|
| Why, not what | Comments explain *why* a decision was made, not *what* the code does (the code itself should be readable enough for *what*) |
| Section headers | Use `# --- Section Name ---` to visually separate major stages within a notebook |
| TODO/FIXME | Never in generated code — if something is incomplete, the spec needs updating, not a TODO in the code |
| Config references | When a value comes from config, comment the config field name: `# from Pipeline_Config.merge_keys` |

## Anti-Patterns (Explicitly Prohibited)

| Anti-Pattern | Why Prohibited |
|---|---|
| Hardcoded table names in transformation logic | Breaks the config-driven principle |
| `try/except: pass` (bare silent exception swallow) | Masks failures; only Logging/Notification components may silently swallow, and they log first |
| Global mutable state | Notebooks are not long-running services; state should flow through function parameters |
| Nested functions more than 2 levels deep | Harms readability; decompose into separate named functions |
| Print statements for logging | Use `Logging_Component` — print statements don't persist and aren't queryable |

## Acceptance Criteria
- [ ] All generated code follows PySpark and SQL conventions listed above consistently.
- [ ] No hardcoded values, magic numbers, or bare string column references.
- [ ] Every function has type hints and a docstring.
- [ ] No TODO/FIXME comments in generated output — gaps must be resolved at the spec level.

## AI Generation Notes
Every agent generating code must treat this document as a hard constraint, not a style preference. Code that violates these standards is a generation failure even if it's functionally correct — consistency across hundreds of generated notebooks is as important as correctness of any single one.

# Secrets Management

**Version:** 1.0
**Last Modified:** 2026-07-13
**Depends On:** Project_Architecture.md (v1.0), CICD_Pipeline_Deployment.md (v1.0)
**Category:** Deployment

## Purpose
Defines how credentials, connection strings, and other sensitive values are stored, accessed, and governed — referenced directly by `SQL_Server_Reader_Component.md` (which mandates secret-scope-based credential retrieval) and by `CICD_Pipeline_Deployment.md` (which separates environments that may have different credentials).

## Scope
Covers Databricks-native secrets management only (secret scopes and the Secrets API). Does NOT cover external vault integrations (e.g., Azure Key Vault) beyond noting them as a future extension — Free Edition's native secret scopes are sufficient for this framework's current scope.

## Where Secrets Live

| Store | What Goes Here |
|---|---|
| Databricks Secret Scopes | SQL Server connection strings, usernames, passwords, any API keys needed by the framework |
| NOT in config tables | Never store credentials in `cfg_source`, `cfg_pipeline`, or any other Delta table — even encrypted |
| NOT in notebooks/code | Never hardcoded, never in comments, never in default parameter values |
| NOT in Git | Never committed, not even in `.env` files or example configs |

## Secret Scope Structure

| Scope Name | Purpose | Example Keys |
|---|---|---|
| `{environment}_sql_server` | SQL Server credentials per environment | `connection_url`, `username`, `password` |
| `{environment}_framework` | Framework-internal secrets (if any) | `notification_webhook_url` |

The `{environment}` prefix (`dev_`, `test_`, `prod_`) matches the environment parameter from `CICD_Pipeline_Deployment.md` — so a notebook running with `environment=prod` reads from `prod_sql_server`, and the same notebook running with `environment=dev` reads from `dev_sql_server`, without any code change.

## How Components Access Secrets

Every Component that needs credentials follows this pattern:

```
secret_value = dbutils.secrets.get(scope=f"{environment}_sql_server", key="password")
```

The `environment` parameter flows from the workflow → notebook → component, and the component resolves the correct scope name from it. No component ever constructs or stores the secret value beyond the immediate use — it's retrieved, used for the connection, and never persisted in any variable beyond that call's scope.

## Implementation Rules
- Secrets are retrieved at the moment of use, not at notebook startup — minimizes the window where a secret sits in memory.
- No secret value may appear in any log, error message, print statement, or audit record — not even partially masked (e.g., `pass***`). The `Logging_Component.md` and `SQL_Server_Reader_Component.md` specs already enforce this, but it's restated here as the authoritative rule.
- Secret scopes must be created manually (one-time setup) by a workspace admin before the framework runs for the first time — this is not an automatable step, and agents should not attempt to create scopes via code.

## Validation Rules
- No generated code may contain a literal credential value.
- No config table may contain a credential column.
- Every secret retrieval in generated code must use `dbutils.secrets.get` with a scope name derived from the `environment` parameter.

## Acceptance Criteria
- [ ] All credentials are accessed exclusively via Databricks secret scopes.
- [ ] Secret scope names incorporate the environment prefix, matching `CICD_Pipeline_Deployment.md`.
- [ ] No secret value appears in logs, config tables, or Git.
- [ ] Secret scopes are documented for manual one-time setup, not generated by agents.

## Dependencies
- `Project_Architecture.md` (v1.0) — establishes Databricks as the platform (where secret scopes exist).
- `CICD_Pipeline_Deployment.md` (v1.0) — environment parameter pattern determines which scope to read from.

## Future Extension Points
- Could integrate with Azure Key Vault (via Databricks' Key Vault-backed secret scopes) if enterprise-grade key rotation and access auditing are needed beyond what native scopes provide.
- Could add secret rotation procedures (how to update a password across environments without downtime) if credential lifecycle management becomes a concern.

## AI Generation Notes
Any agent generating code that accesses external systems must use the secret-scope pattern described here — never generate code with placeholder credentials (e.g., `password = "CHANGEME"`) even as a temporary measure, since those placeholders have a habit of reaching production.
